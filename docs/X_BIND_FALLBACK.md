# X 绑定注册链路预案（grok 自有域直注失效时启用）

> 状态：**预案，未启用**（2026-07-15 固化）  
> 当前自有域直注成功率 ~94%（baoxia.top 0.944 / lima.cc.cd ~0.97），无需动。

## 情报来源

- linux.do [topic 2570427](https://linux.do/t/topic/2570427)：自建域名邮箱直注 grok 走完全流程也可能不可用；
  改走 **先注册 X（Twitter）→ X 绑定注册 grok → SSO 转 CPA** 则正常。
- 互证：[月城帖 2569053](https://linux.do/t/topic/2569053)「协议注册风控史诗级加强」、
  2026-07-15 实测某外部号包 20/20 RT revoked（风控确实在收紧）。

## 触发条件（满足任一即评估启用）

- 自有域某域 24h 滚动成功率 **< 70%**（看注册机日志 `域名健康` 行）；
- 自有域全部域名连续 2 轮 0 成功且失败原因集中在 SSO/风控类（非 mail_code）；
- hotmail 通道同时劣化（无备用通道时优先修通道，不启本预案）。

## 预案链路（高层，启用时再实现）

```
注册 X 账号（邮箱=自有域，同注册机反检测/出口体系）
  → X 账号绑定/授权 grok（grok 侧用 X 登录）
  → 走现有 SSO → CPA 转换链（cpa_xai/protocol_mint.py 不动）
```

要点：

1. X 注册是另一条反检测战线（指纹/手机号/邮箱验证策略不同），**先人工跑通 1 个再谈自动化**。
2. X 账号本身有存活成本（养护/登录态），建议只作 grok 通道的「摆渡」，不当资产囤。
3. 邮箱侧不受影响：自有域收信链路（cloud mail / 云梦 / 备用）原样复用。

## 不启用时的维护

- 每轮注册日志自带 `域名健康` 成功率行，无需额外监控；
- 若启用，本文档升为 runbook，自动化部分走 A2A 工单流程另立。

## 边界

- 不做：X 账号买卖、批量 X 号囤养、绕过 X 的手机号/风控验证。
- 本预案只为保住「自有域邮箱 → grok」的注册能力，不扩大用途。
